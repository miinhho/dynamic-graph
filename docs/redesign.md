# Substrate Redesign

This document captures the new ontology we are committing to after a deep
re-examination of what this engine is for. It supersedes the substrate
implied by phase 1+2 of `architecture.md`, while preserving the *guard rail*
framing established in `identity.md`.

When this document and `architecture.md` disagree, **this document wins**.
When this document and `identity.md` disagree about anything outside of guard
rail framing, **this document wins**. `identity.md` will be updated to fold
this in once the substrate stabilizes.

## 1. Why we are redoing the substrate

Phase 1+2 built a stabilization layer, a regime classifier, and a scheduling
primitive on top of an ontology that was *wrong for the project's actual
goal*. Three implicit assumptions in the old substrate broke under scrutiny:

1. **Entities are primitives.** They were declared upfront and treated as the
   atomic unit. This conflicts with how real systems work — in reality,
   "things" are coherent patterns of relationships, not pre-existing
   substances.
2. **Influence is monolithic.** All effects flowed as a single `SignalVector`.
   Real entities are subject to many *kinds* of influence simultaneously,
   each with different dynamics.
3. **Time is absolute.** `TickId(N)` was a wall-clock-like index. In reality,
   time is the *order of changes*, not a fixed clock that ticks regardless
   of whether anything happens.

The new substrate fixes all three.

## 2. Philosophical commitments

These are the framing decisions that drive the ontology. They are
non-negotiable for this redesign.

### 2.1 Engine is reality-shaped framework

- **Framework**: users bring their own domain. They provide loci, programs,
  influence kinds, perspectives. They consume emergent entities and cohere.
- **Reality-shaped**: the engine's primitives are not arbitrary. They are
  chosen to mirror the structure of real causation — discrete events,
  causal flow, emergent identity, multi-kind influence.
- The framework is *opinionated about how causation works*, but
  *agnostic about what exists in any specific domain*.

### 2.2 Stability is a guard rail, not a goal

(Inherited from `identity.md`.) The engine pushes back only against
divergence. Oscillation, limit cycles, quiescence are valid observation
regimes, not failures to suppress.

### 2.3 Time is the partial order of change

There is no absolute clock. A "batch" is a synchronization barrier — a set
of changes that are causally independent and can therefore be processed
together. Batches are ordered by causal dependency, not by wall time.
Cooldowns, decay rates, and similar concepts must be expressed in
*change-counts*, not seconds.

### 2.4 Entities are emergent, not primitive

This is the deepest commitment of this redesign. Entities are not declared
upfront and stored. They are *recognized* by the engine as coherent patterns
of relationships, which themselves emerge from change-flow. The user provides
the substrate (loci) and the perspective (how to recognize coherence); the
engine produces the entities.

This is process-philosophy / structural-realism / dependent-origination
applied to a graph runtime. Things are patterns. Patterns are real, but
derived.

### 2.5 Causal flow is first-class; cause/effect-as-objects are not

Each `Change` knows its causal predecessors. Walking the change graph
backwards reconstructs any causal chain. But there is no separate `Cause` or
`Effect` type — these are roles inside the change-flow, not standalone
objects.

## 3. The four layers

The new ontology is layered. Each layer derives from the layer below by
some kind of pattern recognition. Each layer is observable.

```
Layer 4: Cohere       — clusters of relationships/entities under a perspective
            ▲                                       (output, derived)
Layer 3: Entity       — coherent bundles of relationships
            ▲                                       (output, derived)
Layer 2: Relationship — observed coupling between loci, derived from change-flow
            ▲                                       (output, derived, entity-like)
Layer 1: Change       — atomic event with causal predecessors
            ▲                                       (primitive, generated by programs)
Layer 0: Locus        — labeled position with state and program
                                                    (primitive, user-registered)
```

### 3.1 Locus (Layer 0)

A `Locus` is the substrate primitive. It is what the user actually registers
with the engine. It is not the same as the user's mental "entity" — it is
just a *position* in the substrate where state lives and changes can happen.

Properties:
- Identity (`LocusId`)
- Kind (`LocusKindId` — a user-defined type tag)
- State (a typed state container — likely still a `StateVector` for
  simplicity, but the kind tag determines its semantic role)
- Program: a function `(current_state, incoming_changes) -> outgoing_changes`
  that the engine calls when changes touch this locus

A locus does not have endpoints. It does not represent a relationship. It
is the simplest possible "stateful position" in the system.

### 3.2 Change (Layer 1)

A `Change` is the atomic event. Everything that happens in the engine is a
change. Changes are first-class — they are recorded, queried, and form the
basis of all higher layers.

Properties:
- Identity (`ChangeId`)
- Subject: the locus that this change affects
- Influence kind (`InfluenceKindId`): what *type* of influence is involved
- Predecessors: a set of `ChangeId`s — the causal antecedents of this change
- Before / after state for the subject locus
- Batch: which causal batch this change belongs to (logical clock index)

A change can be:
- **External** (predecessors empty): a stimulus injected by the user
- **Internal** (predecessors non-empty): produced by a locus program in
  response to incoming changes

The set of all changes forms a DAG ordered by causal predecessors. A
"batch" is a maximal antichain in this DAG — a set of mutually independent
changes that can be processed in parallel.

### 3.3 Relationship (Layer 2, emergent)

A `Relationship` is the first emergent layer. It is not registered by the
user; it is automatically detected by the engine when cross-locus change
flow is observed.

When change A at locus L1 has change B at locus L2 in its predecessors
(meaning A's computation read or was triggered by B's outcome), the engine
recognizes a relationship between L1 and L2 of the relevant kind. The
relationship is *first-class and entity-like*: it has its own ID, its own
state, and its own evolution.

Properties:
- Identity (`RelationshipId`)
- Endpoints: which loci it connects (directed, symmetric, or n-ary)
- Kind (`RelationshipKindId`, often derived from `InfluenceKindId`)
- State: cumulative metrics (activity, recent magnitude, kind distribution)
- Lineage: which changes created and updated it

Relationships are the *substrate for entity emergence*. They are also
themselves valid `Change.subject`s — when a relationship's state evolves,
that itself is a `Change` at the relationship.

### 3.4 Entity (Layer 3, emergent and sedimentary)

An `Entity` is the second emergent layer. It is what the user *cares about*
seeing — the "things" that exist in the system at any moment. But it is
not stored, declared, or registered. It is *recognized* by the engine.

**Key commitment: entities are sedimentary.** An entity is not a snapshot
that gets replaced — it is a *growing record*. Each significant change
deposits a new *layer* on top of the entity's history. Old layers are
preserved (subject to weathering — see §3.6). The "current" entity is
the top of its sediment stack; the entity's *identity* is the entire
stack from Born to now.

The recognition mechanism is:
1. The user provides an `EmergencePerspective` — a function that examines
   the current relationship graph (and existing entity sediments) and
   produces emergence proposals
2. The engine reconciles proposals against existing sediments:
   - High overlap with an existing entity → deposit a new layer on that
     entity (continuation)
   - Partial overlap → could be a *split* (deposit a Split layer on the
     parent, born new entities with parent reference) or a *transformation*
     (deposit a transformation layer)
   - No overlap → born a new entity (Born layer, no parent reference)
3. Existing entities not represented in current proposals:
   - If their members still exist but coherence dropped → deposit a
     `BecameDormant` layer; entity stops accumulating new layers but
     remains in the store
   - If their members merged into a stronger entity → deposit a `Merged`
     layer; the surviving entity gets a `Merged` layer with reference to
     the absorbed ones
4. **Entities are never deleted.** Dormant entities stay in the store
   forever (subject to weathering of their *layers*, not their identity)

Entity lifecycle events are themselves recorded as `Change`s with the
entity as subject. So walking the change graph backwards from any entity
reveals its full history of births, splits, merges, and dormancy.

The user-facing observation surface is primarily this layer. When a user
asks "what entities exist right now?", they get the active (non-dormant)
entities. When they ask "what was entity X like at batch T?", they get
the layer that was top-of-stack at that batch — possibly a compressed or
weathered version (see §3.6).

### 3.5 Memory and weathering (cross-cutting)

Sedimentation alone is unbounded growth. The companion concept is
**weathering** — old records erode over time, in a way that's both
configurable and consistent across the framework.

The framework has four memory layers, each with its own weathering policy:

| Memory layer | What's stored | Default weathering |
|---|---|---|
| **Raw change log** | Every committed `Change` with full detail | Fast: short window of recent batches, full detail inside the window |
| **Relationship state** | Per-relationship cumulative metrics | Continuous: activity decays each batch via the kind's decay rate (already in `InfluenceKind` config) |
| **Entity sediment layers** | Per-entity stack of historical layers | Staged: recent layers preserved in full, mid-age compressed, old removed — except significant transitions (Born/Split/Merged) which resist weathering |
| **Lineage tree** | Parent/child references between entities (just IDs and transition kinds) | Slow: very long retention because cost is tiny |

Each layer's weathering is *pluggable*. The framework ships sensible
defaults so a fresh user can run without configuration, but every layer
can be replaced with a user-provided policy.

Two distinct decay modes coexist:
- **Continuous decay** for live statistics (relationship activity scores).
  Each batch reduces the value by some factor.
- **Staged transformation** for layered records (entity sediment).
  Layers progress through `Preserved → Compressed → Removed` based on age
  thresholds, with significant layers exempted.

Sketch of the weathering API:

```rust
pub trait WeatheringPolicy: Send + Sync {
    type Layer;
    fn weather(&self, layer: &Self::Layer, age_in_batches: u64) -> WeatheringEffect<Self::Layer>;
}

pub enum WeatheringEffect<L> {
    Preserved,
    Compressed(L),
    Removed,
}

pub struct DefaultEntityWeathering {
    pub recent_window: u64,        // age 0..recent_window: untouched
    pub compression_age: u64,      // age recent_window..compression_age: compress
    pub removal_age: u64,          // age >= removal_age: remove (unless preserved transition)
    pub preserved_transitions: SmallVec<[LayerTransitionKind; 4]>,
}

impl Default for DefaultEntityWeathering {
    fn default() -> Self {
        Self {
            recent_window: 50,
            compression_age: 200,
            removal_age: 1000,
            preserved_transitions: smallvec![
                LayerTransitionKind::Born,
                LayerTransitionKind::Split,
                LayerTransitionKind::Merged,
            ],
        }
    }
}
```

Compression strategies (what's lost when a layer is compressed) are
themselves customizable. Default: drop relationship references and
detailed member lists, keep coherence score, member count, and
transition kind.

Even after weathering removes a layer, the **lineage tree** preserves
the parent/child references. So "entity X is descended from entity Y"
remains answerable even if Y's old detail layers are gone.

### 3.6 Cohere (Layer 4, emergent)

`Cohere` is the highest emergent layer. It is *clusters of relationships
and/or entities under a user-supplied perspective*. The user provides a
`CoherePerspective` (a closeness function or clustering algorithm) and the
engine produces the current cohere set.

Multiple perspectives can be active simultaneously, producing different
cohere sets over the same underlying graph.

A cohere is what the user might call "a community", "a context", "a
cultural cluster", "a coordinated group". It is not a primitive — it is a
view computed from a perspective.

## 4. Engine vs user responsibilities

| Engine provides | User provides |
|---|---|
| Locus registry | Locus definitions (id, kind, initial state, program) |
| Change processing pipeline | Locus programs (what changes to produce in response to incoming changes) |
| InfluenceKind registry | InfluenceKind definitions (id, decay defaults, combination rules) |
| Causal batch loop (sync barriers based on causal independence) | Stimuli (root changes injected from outside) |
| Relationship store + auto-emergence | (none — automatic) |
| Entity recognition orchestration | EmergencePerspective (how to recognize coherent bundles) |
| Cohere extraction orchestration | CoherePerspective (closeness / clustering function) |
| Per-kind stabilization (guard rail) | Per-kind stabilization config (alpha, decay, trust region) |
| Per-kind regime classification | (engine has reasonable defaults; user can override) |
| Causal lineage queries (walk change graph) | (engine provides; user consumes) |
| Change log retention (per policy) | Retention policy |

## 6. Tick loop in new framing

The loop is no longer "compute next state for all entities, commit". It is:

```
batch_loop:
  1. Collect changes whose predecessors are all committed
     (these are changes ready to fire in this batch)
  2. For each ready change:
     a. Find subject (locus or relationship)
     b. Run subject's program with incoming
     c. Program produces zero or more new changes
     d. New changes are queued for next batch (or this one if independent)
  3. Apply all ready changes:
     - Update subject states
     - Detect cross-locus causal flow → auto-emerge or update relationships
  4. Run guard rail: per-kind stabilizer on accumulated change magnitudes
  5. Run regime classifier: per-kind, per layer
  6. Commit batch (record all changes that fired)
  7. (Optional, on-demand) Run emergence perspective → produce current Entity set
  8. (Optional, on-demand) Run cohere perspective → produce current Cohere set
  9. Move to next batch
```

Concurrency: changes within a batch are causally independent and can be
processed in parallel. The engine exploits this for parallelism.

## 8. Open questions and resolutions

These are the decisions that shape type signatures. Some are now resolved
(marked **RESOLVED** with the chosen answer); others are deferred to be
decided during implementation (marked **DEFERRED**).

### O1. Change predecessor determination — DEFERRED

- **(a) Automatic via data dependency**: engine tracks read/write sets; if
  change A wrote state X and change B's program read state X, then A is
  in B's predecessors automatically
- **(b) Explicit declaration**: locus programs explicitly name their
  predecessors when producing new changes
- **(c) Hybrid**: automatic by default, explicit override possible

**Tentative direction**: (c) hybrid, default to automatic locus-granularity
read/write tracking, with `ProposedChange::with_extra_predecessors()` for
explicit overrides. Decided during implementation when we can measure the
overhead of automatic tracking on representative workloads.

### O2. Locus program signature — TENTATIVE

Tentative shape (refined during task #15):

```rust
fn process(
    &self,
    locus: &Locus,                       // self
    incoming: &TypedChangeInbox,         // changes grouped by InfluenceKind
    ctx: &ProcessingContext,             // read-only world view + kind registry
) -> Vec<ProposedChange>;
```

- `incoming`: typed inbox grouped by `InfluenceKindId`, each entry holding
  the raw change reference and the projected influence vector
- `ctx`: read-only access to other loci by ID, with auto-tracked reads
  (feeding O1's automatic predecessor mechanism)
- Programs are *pure functions* — same inputs always produce same outputs
- A program can propose changes to:
  - **itself** (most common)
  - **other loci** (allowed but unusual)
  - **NOT relationships** (those are auto-emerged from change flow)

Final TypedInbox API will be settled when we hit it in code.

### O3. EmergencePerspective default vs required — RESOLVED

**Default + override.** The framework ships a default perspective
("connected components in the relationship graph above strength threshold,
sediment-aware reconciliation against existing entities") so a fresh user
gets meaningful entities without configuration. Users can override with a
custom `EmergencePerspective` for richer domain-specific recognition. The
default must be sediment-aware and weathering-aware — i.e., it must use
existing entity sediments when deciding continuation vs split vs new birth.

### O4. Entity lifecycle and identity persistence — RESOLVED

**Sedimentary identity, never deleted.** See §3.4. Each significant
transition deposits a new layer onto the entity's sediment stack. Entity
identity (`EntityId`) is permanent. Reconciliation between current
proposals and existing sediments uses **member overlap** (default ≥ 50%)
combined with **hysteresis** (a borderline overlap must persist for several
batches before splitting/merging is committed) to avoid thrashing on noisy
boundaries. Splits and merges deposit `Split`/`Merged` layers and update
the lineage tree. Dormant entities stay in the store; layer detail
weathers per §3.5.

### O5. Change log retention — RESOLVED

**Pluggable weathering, fast default.** Subsumed into the cross-cutting
weathering concept (§3.5). Raw change log gets a fast-weathering policy
by default (short window, full detail inside). Users plug in custom
`WeatheringPolicy<Change>` for longer retention or richer compression.
Lineage queries that walk beyond the retention horizon return an honest
"trail goes cold here" rather than fabricating data. **Note**: even when
raw changes are weathered away, the entity sediment lineage tree (§3.5)
preserves entity-level history independently — you lose tick-by-tick
detail, not the entity narrative.

### O6. Influence kind: compile-time vs runtime ID — RESOLVED

**Runtime ID** (`InfluenceKindId(u64)`). Consistent with the existing
pattern (`EntityKindId`, `LawId`). Users register kinds in their domain
code via a registry; the engine uses opaque IDs throughout. Type safety
loss is mitigated by making the ID newtype opaque and panicking on
unregistered IDs in debug builds.

### O7. Multi-locus changes — TENTATIVE

**Single-subject only**, with optional `ChangeBundle` for *logical*
grouping of related changes that fired in the same batch. Tentatively
single-subject because:
- Atomicity is conceptually clean
- Multi-locus effects can be expressed as multiple causally-linked changes
- Allowing multi-subject would force atomicity guarantees across loci

Revisit during implementation if a real use case demands joint subjects.

### O8. InfluenceKind vs RelationshipKind — RESOLVED

**Same concept, single ID.** `InfluenceKindId == RelationshipKindId`
(same type or alias). "Thermal" influence creates "thermal" relationships
— they track the same dimension. If a future use case demands sub-kinding
(e.g., "thermal radiation" vs "thermal conduction" as distinct relationship
types unified under "thermal" influence), we add tags or sub-kinds at
that point. Start unified.

### O9. Stimulus and external input — TENTATIVE

**Stimulus = `Change` with empty predecessors.** No separate type. Users
inject via `world.inject_stimulus(proposed_change)` and the engine
schedules into the next batch as a causal root. Simple and consistent
with the rest of the model.

### O10. Concurrency and determinism — TENTATIVE

- **Determinism**: same input changes (with same IDs and contents) → same
  output change graph
- **Within a batch**: changes are causally independent and processed in
  parallel via rayon
- **Conflict rule**: each locus can be the subject of at most one change
  per batch; conflicts force the second change to wait for the next batch
- **ID assignment**: deterministic via content + monotonic counter, with
  intra-batch ordering by `(subject_locus_id, content_hash)` for
  reproducibility
- **Parallelism bound**: rayon's default thread pool, configurable

Final ordering rule decided when we hit it in implementation.

## 9. Out of scope (for this redesign)

- **Persistence / WAL**: change log retention is in scope, but durable
  on-disk persistence is a later concern
- **Network distribution**: single-process only
- **Schema evolution**: how to migrate change logs across substrate version
  changes is not addressed
- **Cross-platform determinism**: deterministic replay is promised on the
  same platform/toolchain only
- **Visual debugging tools**: out of scope, may come later
- **Production-grade performance**: this redesign optimizes for clarity
  and correctness; perf comes after the model is right

## 11. What this commits us to

By accepting this document, we commit to:
1. **Throwing out the current `Entity`, `Channel`, `Emission`, `RecordedDelta`
   abstractions** as primary types
2. **Building a new substrate** based on Locus + Change + emergent layers
3. **Sedimentary entities with weathering** — entities are growing records
   that accumulate layers and erode through configurable decay, never
   deleted, with lineage tree as long-term memory
4. **Cross-cutting weathering policy** for all four memory layers
   (raw change log, relationship state, entity sediment, lineage tree),
   each independently configurable
5. **Porting** stabilizer/regime/adaptive components to per-kind operation
6. **Rewriting** testkit and identity.md to match
7. **Building at least one example domain** to dogfood the framework

This is a substantial rewrite. Phase 1+2 produced ~3400 lines of code. The
rewrite likely produces 1.5x to 2x that, given new layers. The work is
sequenced over tasks 13–22 with explicit checkpoints.

We accept this scope because the alternative — bolting an emergent layer
onto the wrong substrate — would produce a fragile system whose abstractions
fight each other. The user has explicitly committed to "다 갈아엎어야 하면
다 갈아엎어보자" — if we have to tear it all down, let's tear it down.
